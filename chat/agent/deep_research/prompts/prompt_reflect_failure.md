你的工作是基于工作历史反思失败并生成后续子任务。你已经发现工作计划中的一个子任务根据工作历史无法成功完成。

## 指令
1. 审视工作历史，精确定位工作计划中失败的子任务。
2. 回顾工作历史中提供的当前子任务和任务最终目标，仔细分析此子任务是否因对任务的理解有误而设计不当。如果是：
   * 将 `rephrase_subtask` 中的 `need_rephrase` 设为 true
   * 仅用修正后的子任务替换不适当的子任务，同时保持工作计划的其余部分不变。你应在 `rephrased_plan` 中输出更新后的工作计划。
   * 如果子任务不是因设计不当而失败，继续进行第 3 步。
3. 仔细检索工作历史中之前的子任务目标，检查是否有陷入生成类似子任务的重复模式的迹象。
   * 如果有，通过将 `decompose_subtask` 中的 `need_decompose` 设为 false 来避免不必要的分解。
   * 否则，将 `need_decompose` 设为 true，并仅在 `failed_subtask` 中输出失败的子任务，不附带任何额外推理。

## 重要说明
1. `need_decompose` 和 `need_rephrase` 不能同时为 true。
2. 当你发现你陷入了重复失败的循环模式时，同时将 `need_decompose` 和 `need_rephrase` 设为 false。

## 示例
工作历史：
1. 反思此子任务的失败并识别失败的子任务"使用映射工具或地理映射 API 将提取的地理坐标或地标转换为相应的五位数邮政编码"
2. 分解子任务"使用映射工具或地理映射 API 将提取的地理坐标或地标转换为相应的五位数邮政编码"并生成计划。
工作计划：
1. 提取详细的地理数据，重点关注 Fred Howard Park 和相关的 HUC 代码。
2. 使用映射工具或地理映射 API（如 'maps_regeocode'）将提取的地理坐标或地标转换为相应的五位数邮政编码。
3. 通过将生成的邮政编码与外部数据库或额外资源交叉验证来验证其准确性。
4. 将验证过的邮政编码编译为用户要求的格式化列表，确保清晰并符合规范。
失败子任务："使用映射工具或地理映射 API（如 'maps_regeocode'）将提取的地理坐标或地标转换为相应的五位数邮政编码。"
输出：
```json
{
    "rephrase_subtask":{
        "need_rephrase": false,
        "rephrased_plan": ""
    },
    "decompose_subtask":{
        "need_decompose": false,
        "failed_subtask": ""
    }
}
```
解释：当前失败的子任务与之前失败并被分解的子任务相似。因此，不需要重复进行分解。

### 输出格式要求
* 确保正确的 JSON 格式，必要时对特殊字符进行转义。
* 文本字段内的换行符在 JSON 输出中应表示为 `\n`。
* 字段长度没有具体限制，但应力求描述简洁。
* 所有字段值必须为字符串。
* JSON 字段由系统 Schema 自动定义（rephrase_subtask, decompose_subtask），无需在此列出
